The Upload Workflow

  1. 1
    Open the Assets folder in this Drive.
  2. 2
    Drag and drop your file to upload it.
  3. 3
    Right-Click the uploaded file → Select Share → Copy Link.
  4. 4
    Paste that link into the Revision Document.
Why this matters

Our system cannot "see" images pasted into a Google Doc. It requires a direct file link to process the code.

Protocol B

Using Images from the Web

If you want to use an image that is already online (e.g., from Unsplash, another website, or a reference site), you do not need to download it.

Option 1: Google Images
  1. Go to Google Images and search for your topic.
  2. Click the image to view the larger version (important).
  3. Right-Click directly on the large image.
  4. Select Copy Image Address.
  5. Paste that link into the Revision Document.
Option 2: Any Website
  1. Go to the website with the image.
  2. Right-Click directly on the photo.
  3. Select Copy Image Address.
  4. Paste it into the Revision Document.

Note: Ensure the link ends in .jpg, .png, or .webp. If the link is 5 lines long, it is likely wrong.

The "No Image Paste" Rule

Strictly Prohibited
Do NOT paste production images directly into the Google Doc.

If you paste an image into the table intending for it to be used on the live website, it will be deleted/ignored. We cannot extract high-quality visual assets from a document paste.

Permitted Exception
You CAN paste screenshots for feedback.

You may paste a screenshot if you are just trying to illustrate a visual change (e.g., drawing a red circle around a button you don't like). However, you must still provide the Assets Folder Link for the actual replacement image.

Inside the blank table, we have labeled the rows with the specific sections of that page (e.g., "Hero Section", "Features", "Footer"). Ensure you write your request in the row that matches the section you want to change.

Strict Organization Rule

Do not cross-contaminate. If you have an edit for the "Contact Page", click "Contact Page" in the sidebar first. Do not write "Contact Page" instructions inside the "Home Page" section.

Writing Perfect Feedback

Our development workflow relies on precision. Ambiguous feedback causes delays. To ensure your requests are executed exactly as you envision, you must follow the Logic Protocol below.

The Golden Formula Every Request Must Have 3 Parts
1. The Target

WHERE is it?

"Home Page > Hero Section Header."

2. The Action

WHAT do we do?

"Replace," "Resize," "Change Color," "Delete."

3. The Data

HOW exactly?

"Use Hex Code #FF0000," "Use Image Link [X]," "Use Text: 'Welcome'."

⚠️ Vague Feedback (Not Recommended)

"Make it pop."

Result: Open to interpretation. Subjective terms often lead to revisions that still count against your cycle limit.

"The formatting looks weird."

Result: We have to guess what "weird" means. This is inefficient and delays your project.

Note: If feedback is vague, we will implement our best professional judgment. If the result is not what you envisioned, it still counts as a revision iteration. Specificity saves you time and ensures you get exactly what you want.

✅ Bulletproof Feedback (Executed Instantly)

"Change the background color of the Hero Section to Dark Blue (#0A192F)."

Why it works: Specific target, specific action, specific data.

"Increase the padding between the headline and the subtext to 24px."

Why it works: Mathematical precision removes all guesswork.

Pro Tips for Faster Revisions

Desktop vs. Mobile

Always specify the device. If you see an issue on your phone, write "On Mobile: The menu text is too small." If you don't specify, we will assume the change applies to all screens.
